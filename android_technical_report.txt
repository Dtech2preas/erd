DTECH MUSIC - ANDROID TECHNICAL REPORT
========================================

1. OVERVIEW
----------------------------------------
- What the application is: DTECH Music is a native Android music streaming and downloading application built entirely with modern Android development paradigms (Jetpack Compose, Kotlin Coroutines, Room).
- Its main purpose: It provides a seamless, ad-free music listening experience by leveraging backend video/audio services (primarily YouTube via InnerTube and Piped/Invidious APIs) to stream and download music directly to the user's device, maintaining a local SQLite database for library management.
- Key features:
  - Ad-free streaming directly from YouTube sources.
  - Background playback with a fully custom Media3/ExoPlayer integration.
  - Local caching and downloading of audio (via yt-dlp integration).
  - Smart Shuffle (context-aware recommendations using Last.fm and user history).
  - High-End Mode (aggressive pre-fetching of streams for gapless playback).
  - Synchronized Lyrics integration (fetching from LrcLib and KuGou).
  - Robust offline library with custom playlist and favorites management.
  - "Tech Vibe" UI featuring a dark theme with neon blue/gold accents, custom visualizers, and haptic feedback.

2. ARCHITECTURE
----------------------------------------
- Project structure: The app follows a standard Android App module structure (`app/src/main/`).
  - `data/`: Contains Room database definitions, DAOs, and Entity models (`AppDatabase`, `SongDao`, `PlaylistDao`, etc.).
  - `ui/`: Holds all Jetpack Compose screens and UI components (`HomeScreen`, `FullScreenPlayer`, `Navigation`, `Theme`).
  - `workers/`: Background work definitions using Android WorkManager (`MusicDownloadWorker`, `StreamRefresherWorker`).
  - `utils/`: Helper utilities (`SmartShuffleManager`, `LastFmClient`, `RemoteConfigClient`).
  - `lyrics/`: Data providers and parsers for synchronized lyrics.
  - Root package: Contains core business logic and entry points (`MainActivity`, `MusicDownloaderApp`, `MusicViewModel`, `MusicRepository`, `MusicService`, API clients like `InnerTubeClient`).
- How components are organized: The architecture strongly leans towards MVVM (Model-View-ViewModel) combined with Unidirectional Data Flow (UDF) via Jetpack Compose.
- Design patterns used:
  - Singleton Pattern: Used for database access (`AppDatabase.getDatabase`), API clients (`InstanceRegistry`, `InnerTubeClient`).
  - Repository Pattern: `MusicRepository` centralizes data fetching from network (InnerTube/Piped) and local databases (Room), exposing clean APIs to the ViewModel.
  - Observer Pattern: `StateFlow` and `LiveData` in `MusicViewModel` are observed by Jetpack Compose UI components.
  - Factory Pattern: `ExoPlayer` data source factories (`ResolvingDataSource.Factory`).

3. CORE LOGIC
----------------------------------------
- Explain the main functionality of the app step-by-step:
  1. Discovery: Users browse the `HomeScreen`, which populates 'Made for You' and 'Trending' lists by querying InnerTube APIs based on saved genres and favorite artists.
  2. Playback: When a user clicks a song, the `MusicViewModel` requests the `MusicRepository` to fetch the stream URL.
  3. Audio Routing: The stream URL is passed to `MusicService` (which wraps `ExoPlayer`), establishing a foreground media session.
  4. Downloading: Alternatively, users can trigger a download. `MusicDownloadWorker` invokes `YoutubeDL` to fetch the audio file, which is saved locally and registered in the Room database.
- How the app processes data: Data flows from network APIs (JSON parsing via OkHttp) -> `MusicRepository` (mapping and caching) -> Room Database (persistence) -> `MusicViewModel` (StateFlow updates) -> Compose UI (recomposition).
- Key algorithms or workflows:
  - Smart Shuffle Algorithm: `SmartShuffleManager` calculates the next track by looking at session history (to prevent immediate repeats), then querying Last.fm for similar tracks. If that fails, it falls back to the user's top artists, favorites, or genres.
  - Stream Resolution: `MusicService` uses a custom `ResolvingDataSource` to intercept custom URIs (`dtech://stream/ID`) and dynamically resolve them into actual expiring media stream URLs just in time for ExoPlayer buffering.

4. FUNCTIONAL FLOW
----------------------------------------
- What happens when the app starts: `MusicDownloaderApp` initializes the `YoutubeDL` binary environment and schedules the `StreamRefresherWorker`. `MainActivity` sets up the Compose `AppNavigation`. A splash screen (`splash.html` via WebView) hides startup latency while `MusicViewModel` triggers database syncs, loads genre feeds, and refreshes background recommendations.
- User flow from opening the app to using features:
  - Onboarding (if first launch): User selects favorite genres and artists (`GenreSelectionScreen`).
  - Home: User lands on `HomeScreen`, sees personalized feeds.
  - Search/Library: User can search InnerTube (`SearchScreen`) or view downloaded/liked songs (`LibraryScreen`).
  - Player: Tapping a song opens the `MiniPlayer` persistently at the bottom, which expands into `FullScreenPlayer` with lyrics, queue management, and visualizers.
- Background processes:
  - `StreamRefresherWorker`: Runs periodically (every 15 mins) to proactively re-fetch and cache expiring YouTube stream URLs for recommended and library songs.
  - `MusicService`: Maintains a foreground service for uninterrupted background audio playback.

5. UI / UX DESIGN
----------------------------------------
- Layout structure: The app uses a single Activity (`MainActivity`) hosting Jetpack Compose. Screens are logically separate composables, avoiding Android Fragments entirely.
- How the interface is built: Jetpack Compose is used exclusively. UI state is hoisted to `MusicViewModel`. Components are modularized (e.g., `MusicCard`, `SwipeableSongRow`, `DnaComponents`).
- Navigation system: Handled manually via a custom `AppScreen` sealed class and a `BackHandler` stack in `Navigation.kt`, rather than the official Navigation Compose library, allowing for tightly controlled back-stack and state preservation. The design uses a persistent Bottom Navigation Bar and a persistent Mini Player overlay.

6. DATA HANDLING
----------------------------------------
- How data is stored:
  - Local Database: Room SQLite Database (`music_database`) containing tables for `songs`, `playlists`, `play_history`, `favorites`, `stream_cache`, and `lyrics`.
  - Preferences: SharedPreferences managed via `UserPreferences.kt` (settings, genres, high-end mode state).
  - Files: Downloaded `.m4a` files are stored in internal app storage (`filesDir/music_downloads`).
- How data is fetched, processed, and displayed: Network requests use OkHttp. API responses (e.g., from InnerTube client) are parsed from JSON into Kotlin data classes (`VideoItem`), cached via Room if necessary, and then emitted to the UI via Kotlin Coroutines `StateFlow`.

7. DEPENDENCIES
----------------------------------------
- Libraries and frameworks used:
  - Jetpack Compose (UI framework)
  - Kotlin Coroutines & Flow (Asynchronous programming)
  - Room (SQLite object mapping)
  - Media3 / ExoPlayer (Audio playback and media sessions)
  - WorkManager (Background task scheduling)
  - OkHttp (HTTP networking)
  - Coil (Image loading and caching)
  - youtubedl-android (JunkFood02 fork) (YouTube audio extraction)
  - Palette KTX (Color extraction from album art)

8. INTEGRATIONS
----------------------------------------
- External APIs or services used:
  - YouTube InnerTube API (primary source for search, metadata, and streams)
  - Piped / Invidious APIs (fallback instances managed by `InstanceRegistry`)
  - Last.fm API (for similar track recommendations via `LastFmClient`)
  - LrcLib / KuGou (for fetching time-synced `.lrc` lyrics)
  - Remote Config (fetches a `config.json` from GitHub Pages for dynamic API keys).

9. PERFORMANCE & OPTIMIZATION
----------------------------------------
- How efficient the app is: The app is heavily optimized for fast UI rendering. Heavy database operations and network calls are strictly confined to `Dispatchers.IO`. Coil handles aggressive image caching to reduce bandwidth.
- Bottlenecks or improvements:
  - High-End Mode: Actively pre-fetches up to 3 upcoming track stream URLs to guarantee gapless playback, trading network bandwidth for zero buffering latency.
  - Lazy Grids/Rows: The `HomeScreen` and library lists use `LazyRow` and `LazyColumn` for efficient recycling of Compose nodes.
  - `InstanceRegistry` checks multiple Piped/Invidious instances concurrently using `raceInstances` to find the fastest responding node.

10. SECURITY
----------------------------------------
- How user data is handled: All user data (history, libraries, playlists) is stored locally on the device via Room and SharedPreferences. There is no central user account or cloud telemetry.
- Potential vulnerabilities: The app parses raw JSON from various third-party, community-hosted instances (Piped/Invidious) which could theoretically inject malformed data if compromised, though standard JSON parsing mitigates execution risks.

11. BUILD & DEPLOYMENT
----------------------------------------
- How the app is compiled and built: Built using Gradle (Kotlin DSL). It uses Android SDK 34 as the target.
- Requirements to run it: Minimum SDK 24 (Android 7.0). Requires `INTERNET` and `FOREGROUND_SERVICE` permissions. A specific Gradle fix (`jniLibs { useLegacyPackaging = true }`) is required to correctly package the YoutubeDL binaries for specific architectures (armeabi-v7a, arm64-v8a).
